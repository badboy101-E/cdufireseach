# cdufireseach 长期记忆 / 问答缓存设计

副标题：面向“部门位置 / 电话 / 邮箱”类高频问题的 Markdown 记忆库方案

## 1. 背景

当前 `cdufireseach` 的核心场景已经逐渐收敛：

- 用户主要询问成都大学具体部门、科室、办公室的位置
- 也会询问电话、邮箱等高频联系方式
- 这些问题有明显重复性

例如：

- `信息网络中心在哪里？`
- `网络信息中心在哪里？`
- `人事处人事科在哪里？`
- `人事科电话是多少？`

这类问题的特点是：

- 查询目标稳定
- 答案变化频率低
- 用户问法会重复或近似重复
- 每次都重新抓网页和递归判断，成本偏高

因此，需要增加一个**长期记忆 / 问答缓存层**：

- 对重复问题优先直接命中
- 减少不必要的 Firecrawl 抓取
- 提高回答稳定性
- 让人工修正后的高质量答案能够被长期复用

## 2. 目标

新增一个 Markdown 文档作为本项目的长期记忆库，用于：

- 存储高频已验证问答
- 存储标准化问题与答案
- 存储部门位置、电话、邮箱等稳定信息
- 存储来源页与最后核验时间
- 在问题命中时优先直接返回

## 3. 适用范围

优先覆盖以下类型问题：

- `在哪里`
- `办公地点`
- `办公地址`
- `电话`
- `联系电话`
- `邮箱`

优先覆盖以下对象：

- 学校组织机构
- 行政部门
- 直属单位
- 二级站点中的具体科室 / 办公室 / 部门

不建议第一阶段就纳入：

- 新闻动态类问题
- 通知公告类问题
- 时间敏感类问题
- 开放式概况总结类问题

## 5. 记忆库定位

这个 Markdown 文档不只是“FAQ 列表”，而是：

- **高优先级人工确认答案库**
- **稳定字段缓存**
- **长期记忆层**

也就是说，它的优先级应高于实时抓取。

推荐回答优先级：

1. 命中长期记忆库
2. 命中结构化缓存
3. 实时 Firecrawl 抓取 + 智能判断
4. 仍无答案时返回“没有”

## 8. 建议匹配策略

### 8.1 第一层：标准问法精确命中

例如：

- `信息网络中心在哪里？`

直接匹配到已收录标准问法，直接返回答案。

### 8.2 第二层：归一化问法匹配

例如把问题做归一化：

- 去空格
- 去标点
- 去“在哪里 / 地址 / 电话 / 邮箱”等噪音词
- 同义表达统一

例如：

- `网络信息中心在哪里`
- `信息网络中心在哪`

都可映射到同一个记忆条目。

### 8.3 第三层：关键词命中

如果没有命中标准问法，可以根据：

- `对象名`
- `别名`
- `关键词`
- `问题类型`

组合匹配。

例如：

- `人事科电话是多少`

命中：

- 对象：`人事科`
- 类型：`电话`

即可直接从记忆条目中返回电话字段。

## 9. 建议返回策略

当命中长期记忆库时，返回结果仍保持和现有 `ask_cdu` 相同结构，便于上层无感接入：

```json
{
  "answered": true,
  "answer": "办公地点：行政保障中心A328、A330",
  "evidence": "命中长期记忆库：人事科办公地点条目",
  "analysis_steps": [
    "收到问题：人事处人事科在哪里？",
    "命中长期记忆库条目：党委教师工作部（人事处）- 人事科",
    "直接返回已核验答案，不再触发实时抓取。"
  ],
  "matched_site": {
    "name": "党委教师工作部（人事处）",
    "website_url": "https://rsc.cdu.edu.cn/"
  },
  "source_urls": [
    "https://rsc.cdu.edu.cn/jgsz/rsk.htm"
  ]
}
```

这样上层 HiAgent 或 MCP 客户端无需感知答案来自：

- 长期记忆
- 实时抓取

## 10. 与当前系统的集成位置

建议在 `ask_cdu` 的流程里新增一个“记忆命中层”：

```text
用户问题
  -> 先查长期记忆库
  -> 命中则直接返回
  -> 未命中再进入站点定位
  -> Firecrawl 抓取
  -> 智能判断
  -> 返回结果
```

也就是说，新优先级应为：

```text
Memory Markdown
  -> Firecrawl / LangGraph
```

输出：

```json
{
  "matched": true,
```

LangGraph 工作流中可以这样接入：

1. 先查记忆库
2. 命中则直接回答
3. 未命中再走站点解析与抓取

## 12. 更新机制建议

记忆库不能完全手工裸维护，建议采用“人工确认优先”的模式。

### 推荐更新方式

#### 方式 1：人工维护

适合第一阶段。

优点：

- 最稳
- 质量最高
- 可控

#### 方式 2：运行中发现高质量答案后，人工确认再回写

例如系统在实时抓取后得到了高置信度答案：

- `办公地点：行政保障中心A328、A330`

可以先写入候选缓存，再由人工确认后同步进 Markdown。

#### 方式 3：后续升级为结构化数据

如果条目增多，可以从 Markdown 迁移到：

- JSON
- SQLite
- Base / 表格

但第一阶段不建议过早复杂化。

## 15. 结论

这个功能非常适合当前项目，而且优先级很高。

原因是：

- 需求范围明确
- 高重复问题明显
- 位置 / 电话 / 邮箱类问题稳定
- 人工核验后非常适合长期复用

因此建议立即加入一层：

- **Markdown 长期记忆库**

作为：

- 智能体的长期记忆
- 高优先级问答缓存
- 重复问题的稳定答案来源

在未来 LangChain / LangGraph 架构里，这层可以自然升级成：

- 第一个 tool
- 第一个节点

也就是：

- **先查记忆，再查网页**

这会显著提高回答速度、稳定性和最终准确率。
